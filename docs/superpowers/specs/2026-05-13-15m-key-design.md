# 15m-Key — Design Spec

**Date:** 2026-05-13
**Status:** Approved (pending implementation)

## Goal

Theo dõi nến 15m "khóa" của BTCUSDT Futures (USDT-M Perpetual) trên Binance lúc 07:00 giờ Việt Nam mỗi ngày. Sau khi nến đóng, gửi OHLC qua Telegram, rồi giám sát các nến 3m và 5m liền sau, thông báo mỗi khi close của nến phụ vượt HIGH hoặc thủng LOW của nến khóa. Dừng khi mỗi khung đạt 3 break.

## Inputs / Outputs

- **Input:** Streaming klines từ Binance Futures (REST + WebSocket). Cấu hình qua env: `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID`, `SYMBOL` (default `BTCUSDT`).
- **Output:** Tin nhắn Telegram đến `TELEGRAM_CHAT_ID`.

## Khái niệm thời gian

- "07:00 VN" = 00:00 UTC. Nến 15m khóa: 00:00:00 → 00:14:59.999 UTC (đóng tại 00:15:00 UTC = 07:15:00 VN).
- Tool dùng UTC nội bộ để gọi Binance, chỉ format VN khi hiển thị.

## Luồng một ngày

1. **07:15:05 VN** — Scheduler kích hoạt. REST `GET /fapi/v1/klines?symbol=BTCUSDT&interval=15m&limit=2`, chọn nến có `openTime == 00:00 UTC hôm nay`.
2. Gửi Telegram tin "Key candle" với OHLC.
3. Mở 2 WebSocket stream: `btcusdt@kline_3m`, `btcusdt@kline_5m` tới `wss://fstream.binance.com/ws`.
4. Init `count3m = 0`, `count5m = 0`. Lưu `keyHigh`, `keyLow` trong memory.
5. Mỗi event WebSocket có `k.x == true` (nến đóng):
   - `close = parseFloat(k.c)`
   - Nếu `close > keyHigh` → Telegram `📈 {interval} break UP #{count+1}`, tăng counter
   - Nếu `close < keyLow` → Telegram `📉 {interval} break DOWN #{count+1}`, tăng counter
   - Nếu trong range → bỏ qua
   - Nếu counter của khung đó ≥ 3 → đóng WebSocket khung đó
6. Khi cả 2 đóng → log "done", chờ 07:15 hôm sau.

## Kiến trúc — 5 module

| File | Trách nhiệm |
|---|---|
| `src/config.ts` | Đọc & validate env vars |
| `src/binance.ts` | REST fetch klines + WebSocket client (subscribe, on-close-event, reconnect) |
| `src/telegram.ts` | Gửi message qua `https://api.telegram.org/bot<TOKEN>/sendMessage` |
| `src/monitor.ts` | Pure logic: nhận `(keyHigh, keyLow, close, interval)` → `{action: 'break_up' \| 'break_down' \| 'in_range'}`. Quản counter & quyết định khi nào dừng. |
| `src/index.ts` | Wire mọi thứ + scheduler nội (node-cron) trigger 07:15 VN hàng ngày |

## Định dạng Telegram

**Key candle:**
```
🔑 Key 15m BTCUSDT (07:00 VN, DD/MM/YYYY)
O: 62150.0  H: 62380.5
L: 62080.0  C: 62290.1
```

**Break:**
```
📈 3m break UP #1
close 62410.3 > HIGH 62380.5
```
```
📉 5m break DOWN #2
close 62050.7 < LOW 62080.0
```

## Error handling

- **WebSocket disconnect:** reconnect với backoff exponential (1s → 2s → 4s → 8s → max 30s). Sau khi reconnect, tiếp tục với counter hiện tại trong memory.
- **REST fetch fail (key candle):** retry 3 lần cách nhau 5s, sau đó gửi Telegram báo lỗi và bỏ qua ngày hôm đó.
- **Telegram send fail:** retry 1 lần sau 2s, sau đó chỉ log, không crash process.
- **Process restart giữa cửa sổ giám sát:** không persist state — chấp nhận miss thông báo. Khi restart, scheduler vẫn sẽ kích hoạt 07:15 hôm sau bình thường.

## Testing

- **Unit test `monitor.ts`**: cho `keyHigh=100, keyLow=90`, chuỗi close `[95, 101, 102, 89, 95, 103]` → assert đúng action mỗi bước và counter cuối.
- **Manual dry-run mode** (`pnpm dev --dry-run`): fetch nến 15m của *ngày hôm qua* + replay các nến 3m/5m đã hoàn thành trong cửa sổ → in ra console thay vì gửi Telegram. Để verify logic trên dữ liệu thật.

## Out of scope (YAGNI)

- Persist counter qua restart
- Multi-symbol
- Multi-timeframe khóa (chỉ 15m@07:00 VN)
- Backtest GUI hay lưu lịch sử DB
- Báo nến khóa trước khi đóng
